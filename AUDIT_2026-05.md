# Magic v Machine — Mobile-First Full Audit

**Build audited:** v1.7.6 · `main @ a3a7f59` · 2026-05-07
**Scope:** every screen, every system, viewed from six perspectives —
game tester, developer, perf engineer, UX critic, UI critic, marketer
— with a seventh load-bearing axis (balance / S1→S5 difficulty ramp).

This is a working audit, not a launch sign-off. Every finding has a
file:line citation, a severity tier, and a concrete fix. Sections
2 (Critical Fix Queue) and 11 (Recommended Sequencing) are the
bookmarks if you read nothing else.

---

## 1. Executive summary

The game is **shippable** as a premium mobile roguelike. Six
mechanically distinct classes, a Hades-tier Module Fusion system,
sector-themed sub-mechanics, a sub-20-min Challenge gauntlet, and a
sparks-only meta economy already in place. The recent Drop Bay menu,
class emblems, and Challenge Mode rework are launch-quality.

What's holding it back from a polished mobile launch:

- **Accessibility gaps.** `body.reduced-motion` is wired but only
  covers the menu — combat VFX, the death-screen glitch, and the
  520ms staggered menu entrance still run for motion-sensitive
  players. This is a store-listing risk, not just an ergonomics one.
- **Mobile touch ergonomics.** The reroll + end-turn buttons sit
  52×52px under a dice crescent that overlaps the hit zone. WCAG
  44pt fails on 320px devices. First-combat fat-fingers is real.
- **Two perf-tier gates missing.** Combat-recap and turn-digest
  fire full DOM/animation passes per turn regardless of `Perf.tier`.
  On Snapdragon 7-class hardware this costs 8-12ms/turn.
- **Onboarding cost.** ~20 taps from a fresh install to first
  combat (5 storyboard pages + 13 narration steps, no skip).
- **Hidden hooks.** Module Fusions are 12 Hades-style "Duos" that
  ship with no sting SFX, no glyph popup, no tutorial moment. The
  game's most marketable system is invisible in-game.
- **Difficulty curve is mostly fine, with two real bumps:**
  enemy HP jumps 2× from S2→S3 (`constants.js:782-797`) and boss
  effective-DPS jumps 1.71× from S3→S4 (`constants.js:893-912`).
  Both are survivable but feel like cliffs without a relic spike to
  meet them.
- **Save-quota guard absent.** `localStorage` writes are wrapped in
  `try/catch` but exhaustion fails silently. Endless Spire runs at
  Sector 20+ can produce save objects approaching the 5MB quota.

Estimated effort to clear the blocker tier: **~1 day.** Estimated
effort to clear blocker + important: **~3 days.** Polish backlog
beyond that is a 1-2 week ongoing track.

## 3. Performance & frame-budget audit

### 3.1 Render loop & instrumentation

**Healthy infrastructure already in place** (don't rebuild):

- `Game._timers` / `Game._intervals` (`src/game.js:147-179`) —
  tracked timer registry, drained on state transition.
- `Perf` service with tier detection (`'low' | 'mid' | 'high'`) —
  exposed as `Perf.tier`.
- `PerfTrace` for per-frame mark/measure (`src/services/perf-trace.js`).
- `FPSHud` overlay (toggleable) — median frame-time + worst-frame.
- `Diag.dump()` — JSON snapshot of run state for support tickets.

### 3.2 Specific issues

**3.2.1 Music fade timers run outside `_intervals`.**
`src/audio.js:84-125`. `_fadeInterval` uses raw `setInterval(...,
30)`. The fade self-cancels via `_cancelFade()` between calls (so
"timers leak" is overstated), but state transitions
(`Game.clearTrackedTimers`, `gameOver`, `quitRun`) don't drain it.
A fade running into a state that doesn't expect it will keep ticking
for the configured 450-600ms.

**Fix:** Use `Game.setLoop` instead of raw `setInterval`, and clear
the fade in `clearTrackedTimers`.

**3.2.2 Particle pool exhaustion scans linearly.**
`src/effects/particles.js:115-133`. Free-list path is O(1) (correct).
Exhaustion fallback iterates every active particle to find the
shortest-life one to evict. Comment correctly notes "rare path" but
boss-dissolve cascades + reward floating text + screen-flash can
push the pool over capacity for ~200ms.

**Fix:** Drop the eviction strategy entirely — return early (silent
skip) when the pool is full. New spawns over capacity are visual
fluff, not gameplay-load-bearing.

**3.2.3 Touch handlers opt out of passive scrolling.**
`src/game.js:578`. Every button registered via `attachButtonEvent`
adds `touchstart` with `passive: false`. The handler only calls
`e.stopPropagation()` — no `preventDefault()`. The non-passive flag
forces the browser to wait for the JS handler before scrolling,
which on a slow main-thread frame can stall scroll for up to 100ms.

**Fix:** Switch to `passive: true`. Verify dice drag handling
(which DOES need `preventDefault`) is on a different path.

**3.2.4 Combat-recap & turn-digest fire on every turn.**
`src/services/combat-recap.js`, `src/services/turn-digest.js`. Both
build glass panels with layout + animation per turn. No
`Perf.tier === 'low'` early return. On Snapdragon 7 / A12-class
devices this costs 8-12ms/turn — the difference between buttery
turn transitions and noticeably sluggish ones.

**Fix:** `if (Perf.tier === 'low') return;` at the top of `show()`,
or fall back to a single-line static digest with no panel build.

**3.2.5 Death-screen glitch animates `filter:`.**
Per session notes (Day 5), the game-over screen uses
`hue-rotate / invert / saturate` keyframes. Filter animation forces
a full repaint per frame on every device tier. Drops to 30-35fps
on mid hardware, plus the scanline `repeating-linear-gradient`
drift adds a second repaint.

**Fix:** Promote the glitch overlay to its own compositor layer
(`will-change: filter; transform: translateZ(0)`), or animate
`transform: scale()/translate()` instead of `filter`.

**3.2.6 CSS animation surface is large.**
Recent commits added: class-emblem keyframes (8 new animations),
ce-rotate / ce-pulse / ce-orbit / ce-shockwave / ce-blink /
ce-blood-drip. All run simultaneously when emblems are visible.
The reduced-motion guard for these IS in place
(`style.css` class-emblems block — added `body.reduced-motion`
override). ✅

**3.2.7 Service worker cache versioning is manual.**
`service-worker.js:8` and `src/version.js:9`. Both must be bumped
on every shippable commit. Already standard in this repo (we just
bumped to v1.7.6). Risk: a contributor commits without bumping;
returning users get stale assets.

**Fix (deferred polish):** Tie cache key to `git rev-parse HEAD` at
build time, or add a pre-commit hook that auto-bumps.

### 3.3 What's already good

- ✅ Particle pool free-list (O(1) acquire on the hot path).
- ✅ Tracked timer registry (`Game._timers` / `_intervals`).
- ✅ Perf-tier detection + runtime monitor.
- ✅ FPS HUD + PerfTrace marks.
- ✅ DOM-only states exit main loop early.
- ✅ Frame-pace cap on 120Hz displays.
- ✅ `body.perf-low` overrides scattered through stylesheets.
- ✅ Class-emblem animations gated by `reduced-motion`.

## 4. Mobile UX & first-launch audit

### 4.1 First-time launch flow

The path from app-icon to first-combat (no save):

1. App splash / icon → `#screen-start.menu-v2.dropbay`
2. Tap INITIATE RUN → class select
3. Pick class → 5 storyboard pages (TUTORIAL_PAGES,
   `src/constants.js:156-193`): MISSION BRIEFING, INTEL & DATA FILES,
   COMBAT PROTOCOLS, ACTION COMMANDS, PROGRESSION
4. CONTINUE → 13-step combat tutorial (TUTORIAL_NARRATION,
   `src/constants.js:140-154`): "Welcome, Operator", "Your HP keeps
   you alive", … → drag, drop, attack, defend, mana, signature.
5. Tutorial-complete → first real combat

**Tap count: ~20 minimum.** No skip on the storyboard pages. Skip
on the tutorial steps routes to STATE.MAP, but the cost is already
paid because the storyboard came first.

**Fix path:**
- After first run completion, mark `mvm_tutorial_done` in
  localStorage. Subsequent class selects skip storyboard +
  tutorial entirely.
- Add "I've played before" button on the FIRST storyboard page
  (sets the flag, jumps to map).

### 4.2 Touch targets & thumb zones

**Reroll & end-turn buttons.** `style.css:11008-11035`. Mobile
override sizes them at 52×52px, pinned `left: 4px` / `right: 4px`,
`top: 18px`. Dice rise into a crescent (positions computed in JS via
`--arc-x` / `--arc-y` / `--arc-rot`). The leftmost / rightmost dice
overlap the buttons' hit zone by 8-12px depending on dice count.

**WCAG 2.1.** Targets must be ≥44×44 CSS pixels. At 320×568
(iPhone SE first-gen), the cqh-scaling brings buttons closer to
~48×48 — passing minimum but tight. Real complaint isn't size; it's
proximity to dice that respond to taps too.

**Fix:** Bump to 56×56 on mobile, add a 4-6px padding ring of
"dead zone" around each, and consider a translucent panel
background behind both buttons so they read as "the controls" not
"another die".

### 4.3 Information density

The combat HUD top-bar (`index.html:790-810`) carries: sector
display, sector mech pill, modules dropdown, loadout button, combat
log button, settings button. On 360px-wide phones with the recent
operator-chip removal, this row is now reasonable. The recent move
of CHALLENGE + PACT pills under sector-display (commit `bc8dc1a`)
fixed a real overflow.

**Map screen** — the legend at the top has 6 node types in normal
runs (combat / elite / event / shop / rest / boss) + 3 in challenge
(rest / boss / pact). At 320px the legend can wrap awkwardly.
**Fix (polish):** swap to icon-only on viewports <360px with a
long-press tooltip for the label.

### 4.4 Tooltip behaviour

`src/ui/tooltip.js:26-27` — clamps left/right to 10px from
viewport edge. **No `max-width` enforced.** A long pact / synergy
description (e.g. *PACT OF SILENCE — Rerolls cost nothing. Combo
bonuses no longer trigger.*) at 360px can exceed the available
340px and crowd the edge or wrap unattractively.

**Fix:** Add `max-width: min(280px, calc(100vw - 24px))` to the
tooltip element.

### 4.5 Modal & overlay behaviour

Modals: `loadout-modal`, `save-slot-modal`, `custom-run-modal`,
`char-detail-overlay`, `info-overlay`, `modal-ascension`,
`screen-pact`, `screen-module-pick`. Most reparent themselves to
`<body>` on first open (defensive — `src/game.js:9844-9851` for
loadout) to dodge containing-block traps from `will-change` /
`filter` / `transform` ancestors.

**Audit task:** find every ancestor of every modal that has those
properties; either strip them or confirm the reparent runs before
first use.

### 4.6 Transitions

The `.menu-v2.dropbay.screen.active` stagger-in
(`src/styles/menu.css:1110-1129`) runs 520ms cubic-bezier with
80-780ms cascading delays. **Not guarded by reduced-motion.** For
players with motion sensitivity, this reads as lag, not delight.

**Fix:** Add a `body.reduced-motion .menu-v2.dropbay.screen.active *`
override that sets `animation: none !important;`.

## 5. UI & visual-language audit

### 5.1 Typography

5 font families in use: Orbitron (display), Saira (impact),
Audiowide (tech eyebrows), Rajdhani (body), system fallback. This
is one too many — Saira and Audiowide overlap in role. At small
sizes Audiowide's 0.18em letter-spacing degrades readability.

**Fix (polish):** consolidate to 3 — Orbitron (titles), Rajdhani
(body), system (everywhere else). Or document the role of each
in `src/styles/tokens.css` so the choice is intentional, not
accumulated.

### 5.2 Color tokens

Tokens defined in `src/styles/tokens.css` (--ui-brand, --ui-alert,
--ui-gold, etc.). But ~77 raw hex codes leak past the token system
across `style.css`, `menu.css`, `bracket-screens.css`. Sector
config colors (`menu.css:54-62`) are all hardcoded.

**Fix (polish, multi-session):** alias each leaked hex to a token
or document why it's an exception (e.g. sector palette intentionally
diverges from brand palette).

### 5.3 Layout failure modes

- ✅ Settings gear top-right — fixed in v1.7.6 (`src/styles/menu.css`
  `#screen-start.menu-v2.dropbay #btn-settings-main.db-gear-floating`).
- ✅ Streak/version footer flush to canvas bottom — fixed via
  `.db-bottom-block { margin-top: auto }` (v1.7.6).
- ✅ Module picker no longer clips — `#screen-module-pick` flex-start
  (commit `c3e73a4`).
- ⚠ iOS gesture zone vs dice tray — bottom-controls padding uses
  `safe-area-inset-bottom` correctly (`style.css:11016`) but no
  real-device validation of iPhone 12+ home indicator collision.
  **Action:** `mobile-qa.md` row to verify on hardware.
- ⚠ 320px width (iPhone SE) — dice are 56×56 + spacing; can the
  full crescent fit? Untested on 320px.

### 5.4 Visual consistency

Recent additions feel cohesive:

- ✅ Class emblems (6) follow consistent 200×200 viewBox + colour-
  per-class + animation grammar (`src/ui/class-emblems.js`).
- ✅ Module-pick screen reuses `.reward-card` markup so it reads
  identical to post-combat reward (`src/game.js _openModulePicker`).
- ✅ Pact picker + Challenge map share the cyan/red theme correctly.
- ✅ Drop Bay menu has a coherent grid + ops cells + console + tabs
  + footer flow.

What's inconsistent:

- 5 fonts (above).
- Heading sizes vary screen to screen (`#screen-rest h2` vs
  `#screen-pact h2` vs `#screen-reward h2` — recent fixes scale
  some down to 1.5rem but not all). **Fix:** tokenise heading
  scale.
- Screen entry animation is tightly tuned for menu (520ms stagger)
  but other screens use the base `.screen { transition: opacity
  420ms ... }`. Feels different.

### 5.5 What looks generic

The neon-cyan-pink-gold palette is well-executed but commonplace
in cyberpunk. The standout element is the **per-sector backdrops**
(`style.css:13986-14000+`) — each sector has a distinct grid colour
+ ambient mood. **Lead with these in store screenshots.** The
default menu cyan + the Sector-3 magma backdrop next to each other
sells "this is a journey, not one mood".

## 6. Accessibility audit

### 6.1 Reduced motion

**Coverage check:**

- ✅ `class-emblem` animations gated (`style.css` recent block).
- ✅ Some menu animations (`src/styles/menu.css:1054-1060`).
- ❌ Combat VFX (screen flash, shake, particle bursts) — no global
  `body.reduced-motion` guard. Each VFX call needs to check.
- ❌ Death-screen glitch (filter keyframes) — runs unconditionally.
- ❌ Menu stagger-in (520ms cascade, `menu.css:1110-1129`) — runs
  for everyone.
- ❌ Reroll-pulse (`style.css:3351-3355`) — no guard.
- ❌ End-turn no-moves animation (`style.css:4989`) — no guard.
- ❌ Map node entry stagger (`renderMap` in game.js, sets
  `setTimeout` to remove `node-entering`) — runs unconditionally.

**Fix:** add a global rule to `style.css`:

```css
body.reduced-motion *,
body.reduced-motion *::before,
body.reduced-motion *::after {
    animation-duration: 0.01ms !important;
    animation-iteration-count: 1 !important;
    transition-duration: 0.01ms !important;
}
```

…then audit any VFX that uses `requestAnimationFrame` rather than
CSS to add an explicit `if (body.classList.contains('reduced-motion'))`
short-circuit.

### 6.2 Text scale

`src/styles/tokens.css` has a `--ui-text-multiplier` token. **No
UI control to change it.** Players who need larger text fall back
to browser zoom (which can break layouts). For an App Store-listed
game, accessibility-control absence is a soft compliance flag.

**Fix:** add a slider in the settings modal — three values
(100% / 115% / 130%) is enough.

### 6.3 Dyslexia-friendly font

Mentioned in roadmap, not implemented. **Fix:** add a settings
toggle that swaps `--font-body` to `Atkinson Hyperlegible`
(open-source, similar metrics to Rajdhani). Defer the font
download until toggled.

### 6.4 Colorblind safety

`Palette.setMode(mode)` is wired from `sel-colorblind` settings
dropdown (`src/game.js:954`). Rarity badges use symbols (◆ ★ ✦ ☣)
on reward + module pick cards — symbol fallback works for
deuteranopia / protanopia. ✅

What's NOT enforced:

- Map node types are **icon-only** in the legend but **colour-coded**
  in the node bubbles (red boss, green rest, gold shop). For a
  monochrome-vision player the icon SVGs DO carry the info, but
  the node type pills around them rely on hue.
- Synergy hints in reward cards use color alone for "ready" vs
  "needs more". Add a glyph (⚡ vs ◇ — glyphs ARE there. ✅).

Verdict: colorblind mode is well-handled. Minor polish on map
node colour vs symbol redundancy.

### 6.5 Contrast

Anything dark-on-dark in the UI? Spot checks:

- Reward card body text (`--text-muted`, ~rgba(255,255,255,0.6))
  on rgba(8,12,24,0.72) — ratio ~4.5:1. Borderline AA.
- Sector-mech-pill at opacity 0.9 on dark — fine.
- Combat tooltips — depends on tooltip background, which is set
  inline. Worth a contrast pass.

## 7. Game-design / tester audit

### 7.1 Run flow

Standard run (5 sectors × ~6-8 nodes each with a forced rest at
layer 1 + 2, shop at 5, elite at 7, rest at 9-10, boss at 11) =
**~30-40 fights, ~30-60 minutes**. Mobile-appropriate. ✅

Challenge Mode = 5 bosses + 10 rests (2 per boss) + 1 pact node =
17 nodes. Sub-20-min arcade gauntlet. ✅

Archive Mode = 1 fight (Archivist). End-game pinnacle. ✅

### 7.2 Soft-lock risks

**Reward screen empty pool fallback.** `src/game.js:21059-21080`
handles the case where every relic is owned/maxed: gives 25 frags
consolation, completes node, returns to map. ✅

**Module picker empty pool.** Same fallback in `_openModulePicker`
(recently added). ✅

**Pact picker.** Forced commit (no cancel) on the Challenge run-
start node. If `activePacts.size === 3` the picker would render
with all signed cards and no clickable target. **Edge case:**
since the pact node runs FIRST (before any rest), `activePacts` is
always empty there. Safe by construction. ✅

**Map traversal.** Iced edges (S2) and glitched edges (S5) gate
travel. Iced edges block until a Heat Vent event fires; if no
event spawns the player has no path. **Edge case:** the iced-edge
generation excludes "edges whose target node has only one inbound"
(`src/game.js:8845`), so a fully-iced run shouldn't be possible.
✅ but worth a stress test.

### 7.3 Save robustness

`saveGame()` writes JSON to `mvm_save_{slot}` in localStorage.
Migrations run on `loadGame()`. If `localStorage.setItem` throws
(quota exceeded / disabled), the catch is silent — the run state
is held in memory until the player force-quits, then lost.

**Fix:** When `saveGame` catches a write error, surface a toast
("STORAGE FULL — DELETE OLD SLOTS?" with a button to open the
slot picker). Auto-prune the oldest non-active slot if save fails
twice in a row.

### 7.4 Content depth

**Verified counts (from `src/constants.js`):**

| Content | Count |
|---------|-------|
| Player classes | 6 |
| Enemies (across sectors) | ~50 unique entries |
| Bosses | 5 + Archivist |
| Events | ~25+ in EVENTS_DB |
| Modules (UPGRADES_POOL) | ~70+ |
| Module Fusions | 12 |
| Synergies | ~12 |
| Corrupted Relics | ~10 (gated by Ascension) |
| Sparks upgrades | ~12 (Sanctuary) |
| Meta upgrades | ~14 |
| Ascension tiers | 0-20 |
| Sector mechanics | 5 (one per sector) |
| Custom Run modifiers | 15 |
| Pacts | 3 |

**Verdict:** depth is solid for a premium roguelike. Not Slay-the-
Spire-deep but not pretending to be.

### 7.5 Mode overlap

**Custom Runs vs Challenge Mode** — Custom Runs lets players
toggle 15 modifiers; Challenge Mode is a fixed gauntlet. From a
player's perspective, "make my run harder for more reward" already
exists as Ascension. So Custom Runs occupies an awkward middle
slot: not as committed as Ascension, not as fixed as Challenge.

**Recommendation:** keep Custom Runs but hide behind an "Advanced"
toggle in the menu cluster until usage data shows >25% of the
playerbase opens it. Don't market it as a headline feature.

### 7.6 Endless Spire (post-Sector-5)

`s_endless` Sanctuary upgrade unlocks an endless mode. Sectors 6+
reuse the Sector-5 backdrop. No new bosses past Tesseract Prime.
HP scales linearly per endless tier.

**Verdict:** thin endgame. Either:
- Cycle 3-5 backdrop variants (`endless % 5` pattern, reuses
  existing `playSectorIntro` branches), or
- Label the feature "Early Access" / "More coming soon" so players
  don't expect S6-S10 to be fundamentally different.

## 8. Balance & difficulty-ramp audit (load-bearing)

User flagged this as "very important". I read the actual numbers
in `src/constants.js`. Findings differ from the explore-agent
guess.

### 8.1 Enemy HP curve

Per-sector mean HP from `ENEMIES` (`src/constants.js:772-870`):

| Sector | Mean HP | Ratio vs prior |
|--------|---------|---------------|
| 1 | ~38 | — |
| 2 | ~55 | **1.45×** |
| 3 | ~110 | **2.00× ← steepest** |
| 4 | ~125 | 1.13× |
| 5 | ~175 | 1.40× |

The S2→S3 jump is the steepest by far. This matches the original
"sector 3 is brutal" complaint that's surfaced in playtest notes.

### 8.2 Enemy DMG curve

| Sector | Mean DMG | Ratio vs prior |
|--------|----------|---------------|
| 1 | ~7 | — |
| 2 | ~11 | 1.57× |
| 3 | ~14 | 1.27× |
| 4 | ~15 | 1.07× ← flat |
| 5 | ~26 | **1.73× ← steepest** |

S3→S4 is **flat** in damage despite the visual sector change.
S4→S5 is the damage cliff.

### 8.3 Boss HP and effective DPS

`BOSS_DATA` (`src/constants.js:872-922`):

| Sector | HP | DMG | Acts/turn | Effective DPS |
|--------|-----|-----|-----------|---------------|
| 1 | 300 | 20 | 2 | 40 |
| 2 | 500 | 40 | 1 | 40 (+ Void pressure) |
| 3 | 750 | 70 | 1 | 70 |
| 4 | 900 | 30 | 4 | **120 ← spike** |
| 5 | 1000 | 50 | 3 | 150 |

S3→S4 boss DPS jumps **1.71×** (70→120). The Hive Protocol's
4-actions-per-turn is the load-bearing element.

### 8.4 Player power scaling

Players earn ~1-2 relics per sector (3 reward picks per fight,
several fights per sector, but also stacking caps). Sparks +
Ascension scaling are largely flat per sector.

This means **the player's power curve is sub-linear** vs enemy HP
S2→S3 (2×) and boss DPS S3→S4 (1.71×). The runs that fail mostly
fail at these two cliffs.

### 8.5 Recommended balance amendments (one number per commit)

I will make these changes in the Balance pass, with a rationale
and before/after numbers in each commit body. Tentative direction:

- **Smooth S2→S3 enemy HP.** Reduce S3 enemy mean HP by ~15%
  (e.g. Magma Construct 100→85, Core Guardian 120→100, Foundry
  Golem 140→120). Keeps the cliff but makes it climbable.
- **Smooth S3→S4 boss DPS.** Either reduce Hive Protocol
  actions/turn 4→3 (DPS 120→90) or reduce damage 30→24 (DPS
  120→96). Per-action damage drop preserves the "many fast hits"
  fantasy.
- **Boost S4 enemy damage.** S3→S4 enemy DMG is flat (1.07×). Bump
  S4 mean dmg by ~15% (e.g. Praetorian 15→17, Sentinel Orb 20→23,
  Phase Stalker 25→28). Restores the curve without spiking the
  boss.
- **Verify Ascension multiplicative stacking.** Read `_ascEffects`
  application points; if Ascension 5+ stacks 1.2× on enemy HP and
  1.2× on enemy DMG simultaneously (1.44× compound), document that
  in the Ascension picker UI. Currently players don't see the
  compound math.

### 8.6 Challenge Mode reward parity

The 3× sparks reward on Challenge final boss is a flat multiplier
(commit history: shipped earlier this sprint). Normal-run sparks
math at A0 = 3 + 4×2 + 15 = ~26 sparks for a Sector 5 clear.
Challenge final = 3× the boss-kill + win-asc grants = ~78 sparks.

Ascension 5+ adds ~+15 sparks on the win bonus. Challenge
multiplies that too (+45). At Ascension 20 the Challenge run
gives ~150 sparks vs normal's ~50. **This is fine** — Challenge is
arcade mode, the 3× bonus is the reward for committing to it.

### 8.7 Custom Run payout incentive bug

`CUSTOM_RUN_MODIFIERS` (`src/constants.js:1641-1657`) has each
modifier carrying its own `payoutBonus` (e.g. Locked In = +35%,
Glass Cannon = +20%). The `_applyCustomRunModifiers` should sum
these. Verify it does — if it picks the max instead of summing,
players who select multiple negatives are paid less than they
should be, which is a feel-bad. **Action:** read the apply code
and confirm.

## 10. Architecture & code-quality audit

### 10.1 `src/game.js` is monolithic

~30k lines. Every state, every UI, most rendering. Has been
deliberately kept this way — the top of file (`src/game.js:1-25`)
imports from sub-modules cleanly. The monolith is a maintenance
issue but not a perf one.

**No action recommended for this sprint.** A future refactor could
extract: combat loop, map render, modal layer. Each chunk ~3-4k
lines and individually testable.

### 10.2 Silent error swallow

`try { ... } catch (_) {}` patterns scattered through `game.js`.
Many are intentional (defensive around `localStorage`,
`navigator.vibrate`, `AudioContext.resume`). Some are not — search
for `} catch (_) {}` (truly empty) shows ~30 sites.

**Fix (polish):** convert `catch (_) {}` to
`catch (e) { console.warn('[context]', e); }` on a sweep so
production failures surface in the diag dump.

### 10.3 Dead code from recent commits

Recent commits added a lot. Audit pass:

- ✅ `_paintCharPreview` — still called for char select (kept).
- ✅ `.db-hdr` CSS rules in menu.css — element removed, rules
  remain. Harmless but dead.
- ✅ `.db-spacer` — element exists in HTML, hidden by CSS
  `display: none`. Could be removed entirely from HTML.
- ✅ `#turn-display` — removed from index.html, JS lookup
  no-ops. Clean.
- ✅ `#intel-profile` — removed from HTML, JS `setText` calls
  no-op via getElementById guard. Clean.

**Sweep recommendation:** delete `.db-hdr` rules + `.db-spacer`
HTML element when convenient. Cosmetic, zero risk.

### 10.4 Stylesheet specificity wars

Recent commits hit specificity-collision three times (settings
gear, operator chip, footer flush). The pattern: `polish.css`
loads after `menu.css` and has higher-specificity selectors that
beat `menu.css` rules.

**Mitigation already established:** new menu.css rules use
`#screen-start.menu-v2.dropbay <descendant>` form to push
specificity above polish.css. Document this convention in a
header comment in `menu.css` so future contributors don't
re-litigate.
